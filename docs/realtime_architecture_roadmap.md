# ARBATOS 实时性优先架构路线

结论先写在前面：ARBATOS 后续升级要先保实时性，再谈框架高级感。高频控制链要收紧成一条边界清楚、耗时可测、行为可预测的快路径；日志、配置、上位机、裁判系统、调试、模块管理这些功能只能放在慢路径。

这份文档的目标不是马上推翻现有代码，而是给下一步改造定边界、定顺序、定验收标准。

## 1. 现在最重要的问题

当前代码已经有比较清楚的任务划分，例如：

- `shared/application/gimbal_control_task.c`：云台和射击高频控制。
- `shared/application/chassis_control_task.c`：底盘控制。
- `shared/application/can_feedback_rx_task.c`：CAN 电机反馈接收。
- `shared/application/can_command_tx_task.c`：CAN 电机指令发送。
- `shared/application/rc_sbus_task.c`：遥控输入。
- `shared/application/sdlog.c` / `sdlog_task.c`：SD 卡日志。
- `shared/application/watch.c`：运行状态观察。

这种“一个任务管一块”的写法能让结构清楚，但不等于实时性天然可靠。实时性真正怕的是这些东西混进高频链路：

- 每 1ms / 2ms 都写日志或做复杂状态统计。
- 高频任务里读很多全局配置，导致控制链耗时不稳定。
- 高频任务之间靠调度器轮流抢时间，顺序不完全可控。
- CAN 接收任务一次处理所有帧，没有明确时间上限。
- 控制任务里出现文件系统、打印、动态内存、复杂消息分发、长临界区。

所以后续改造的核心不是“加更多框架”，而是先把高频链路变窄、变硬、变可测。

## 2. 参考项目怎么学

这些项目值得参考，但要学它们的实时性边界，不要只学外表。

| 项目 | 值得学什么 | 对 ARBATOS 的启发 |
|---|---|---|
| Betaflight | 陀螺仪、滤波、PID 优先，其他功能吃剩余时间 | 云台、射击、CAN 发送应当像它的快循环一样被保护 |
| ArduPilot | 任务表里写频率、预计耗时、优先级 | ARBATOS 每个任务也应该有预算和超时统计 |
| PX4 | 飞控主功能和中间层分开 | 控制、估计、驱动、通信要分边界，但不要把消息系统塞进快路径 |
| VESC | 电机控制靠硬件时序和中断保护 | 最要命的控制环尽量靠近硬件，慢功能放外面 |

## 3. 快路径和慢路径

### 快路径

快路径就是必须准时完成的控制链。这里宁可朴素，也不要花。

建议放进快路径的东西：

- IMU 采样或最新姿态快照。
- CAN 电机反馈快照。
- 云台 1kHz 控制。
- 射击 1kHz 控制。
- 底盘 500Hz 控制。
- CAN 电机命令统一发送。
- 极少量耗时统计，例如最大耗时、超时次数。

快路径禁止放这些东西：

- `malloc` / `free`。
- 文件系统和 SD 卡直接写入。
- `printf` 或任何阻塞式输出。
- 复杂消息总线。
- 动态模块遍历。
- 等锁、等队列、等信号量。
- 没有上限的 `while` 循环。
- 大段临界区。
- 高频读取大量全局配置。

### 慢路径

慢路径可以写得更优雅，也可以吸收 `librm / SRML / UIML` 这类项目的思想。

适合放在慢路径的东西：

- 上位机通信。
- 视觉协议解析。
- 裁判系统解析。
- 遥控输入源选择和输入融合。
- 参数更新。
- 日志写入。
- 状态灯、蜂鸣器、调试观察。
- 设备在线检测。
- 模块注册和模块管理。
- 低频事件通知。

慢路径可以更模块化，但只能向快路径提供“已经整理好的输入快照”或“低频参数更新”。它不能直接插进 1ms 控制循环。

## 4. 建议的最终运行结构

第一阶段不一定马上改成这样，但最终方向建议如下：

```text
中断 / DMA
  |
  +-- CAN RX: 只收帧，尽快更新电机反馈快照
  +-- IMU: 只更新姿态或原始采样快照
  +-- UART / USB: 只收数据，解析放慢路径

高优先级快路径
  |
  +-- control_executor_task 1kHz
        |
        +-- 读取输入快照
        +-- 读取姿态和电机反馈快照
        +-- 云台控制
        +-- 射击控制
        +-- 每 2 次执行一次底盘控制
        +-- 统一生成 actuator_cmd
        +-- 统一发送 CAN 命令
        +-- 记录极简耗时统计

普通优先级慢路径
  |
  +-- host_link_task
  +-- referee_rx_task
  +-- rc_sbus_task / elrs_task
  +-- detect_task
  +-- config_service
  +-- diagnostics_service

低优先级后台
  |
  +-- sdlog_task
  +-- debug dump
  +-- 参数保存
```

这个结构的重点是：控制顺序固定，CAN 发送跟着控制结果走，不再完全依赖多个高优先级任务之间的调度运气。

## 5. 分阶段实施路线

### 阶段 0：先测量，不先重构

先加一个很小的实时性统计模块，例如：

- `shared/application/rt_profiler.h`
- `shared/application/rt_profiler.c`

它只做几件事：

- 记录任务本次耗时。
- 记录最大耗时。
- 记录平均耗时或滑动平均。
- 记录超过预算的次数。
- 记录最长临界区耗时，如果后面愿意加。

推荐用 DWT 计数器或硬件定时器。不要在 1ms 任务里直接打印，也不要每圈写 SD 卡。

第一批要测的点：

- `gimbal_control_task` 整圈耗时。
- `chassis_control_task` 整圈耗时。
- `can_command_tx_task` 整圈耗时。
- `can_feedback_rx_task` 单次被唤醒后的处理耗时。
- `sdlog_write` 在高频任务里单次耗时。
- `watch_task_beat` 在高频任务里单次耗时。

验收标准先不要定死，但建议先用这些目标：

- 1ms 快路径最坏耗时不超过 700us。
- 2ms 底盘路径最坏耗时不超过 1200us。
- 正常比赛负载下，10 分钟内高频任务超时次数为 0。
- 高频任务里单个临界区尽量低于 10us。
- CAN 接收突发处理不能长时间霸占高优先级。

### 阶段 1：不改大架构，先修明显抖动点

这一阶段只做低风险改动。

建议改动：

- 把 `chassis_control_task` 从 `vTaskDelay` 改为 `vTaskDelayUntil`。
- 检查 `gimbal_control_task` 里重复调用 `gimbal_set_control` 是否必要，不必要就删掉一次。
- 把 `can_command_tx_task` 的周期从“跟着底盘周期”改成独立配置，云台和射击如果需要 1ms 命令，就不要被底盘 2ms 拖慢。
- 高频任务里只记录极少量统计，详细日志交给 `sdlog_task` 低频写。
- `watch_task_beat` 在高频任务里只做计数，不做复杂整理。
- CAN 接收任务处理队列时加时间预算，或者每次最多处理固定数量帧，避免极端情况下霸占 CPU。
- 高频任务不要每圈读一大坨 `g_config`，把常用参数做成本地快照，慢路径更新后再安全切换。

这一阶段的目标是让现有结构先稳定下来，不追求马上变高级。

### 阶段 2：确定快路径接口

下一步要把快路径用到的数据整理成几个明确结构。

建议准备这些概念：

- `sensor_snapshot_t`：姿态、角速度、IMU 状态。
- `motor_feedback_snapshot_t`：电机反馈快照。
- `input_snapshot_t`：已经处理好的遥控、图传、上位机输入。
- `control_config_snapshot_t`：高频控制只读参数快照。
- `actuator_cmd_snapshot_t`：本周期要发给电机的命令。

原则：

- 快路径只读快照，不直接关心协议细节。
- 慢路径负责把协议、配置、遥控、裁判系统整理成快照。
- 快照切换要短临界区完成，不能在高频任务里做复杂复制。
- 快路径输出统一进 `actuator_cmd` 或新的命令快照。

这样做的好处是：后面你想引入更高级的模块系统，也只能影响慢路径和快照生产，不会污染控制链。

### 阶段 3：尝试控制执行器

当阶段 0 和阶段 1 的数据说明当前多任务调度仍然抖动明显时，再考虑做 `control_executor_task`。

建议执行顺序：

```text
1ms tick:
  read snapshots
  run gimbal control
  run shoot control
  if tick % 2 == 0:
      run chassis control
  pack actuator commands
  send CAN commands
  update profiler counters
```

注意点：

- 这不是把所有代码硬塞到一个巨型文件里。
- 云台、底盘、射击仍然可以保留各自模块。
- 只是把它们的运行顺序收归一个高频执行入口。
- CAN 发送最好跟控制结果在同一条链路里，减少“算完了但还没发”的时间差。

如果暂时不想大改，也可以先做半步：

- 保留 `gimbal_control_task` 和 `chassis_control_task`。
- 先把 `can_command_tx_task` 提升为固定 1ms。
- 控制任务只更新命令快照，CAN 任务每 1ms 发送最新快照。

但从实时性角度看，最终还是统一高频执行入口更容易分析。

### 阶段 4：把高级框架思想放进慢路径

等快路径稳定后，再融合 `librm / SRML / UIML` 这类项目的优点。

可以融合的方向：

- 模块注册：让慢路径模块有统一初始化、启动、停止、状态上报入口。
- 配置服务：统一管理参数更新、参数校验、参数快照切换。
- 事件通知：给低频模块用，例如设备上线、模式切换、错误上报。
- 诊断中心：集中收集任务耗时、设备在线、错误码、电源状态。
- 协议适配层：上位机、视觉、裁判系统、遥控输入都变成输入源，不直接碰控制器内部状态。

不要融合的方向：

- 不要在 1ms 控制循环里走消息总线。
- 不要在快路径里做模块查找。
- 不要在快路径里动态注册或反注册模块。
- 不要让日志系统直接参与控制链。
- 不要让配置系统在控制循环中做复杂解析。

人话就是：高级框架可以管“谁存在、谁在线、谁发了什么低频事件”，但不能管“这一毫秒电机该给多少电流”。

## 6. 推荐优先级模型

这只是建议，实际还要按测量结果调。

| 层级 | 内容 | 要求 |
|---|---|---|
| 中断 | CAN RX、IMU、UART DMA 半满/满中断 | 只搬数据和通知任务，不做复杂计算 |
| 最高优先级 | IMU 融合或控制执行器 | 固定周期，不能阻塞 |
| 高优先级 | CAN 反馈快照处理、CAN 命令发送 | 时间有上限，不能无限处理 |
| 普通优先级 | 遥控、上位机、裁判系统、设备检测 | 可以模块化，但不能影响快路径 |
| 低优先级 | SD 日志、调试输出、统计整理 | 只能吃空闲时间 |

如果后面使用统一 `control_executor_task`，它应该成为最重要的高频入口。其他任务都要围绕它提供快照或消费结果。

## 7. 代码规则

后续可以把这些规则当成约束。

快路径规则：

- 必须有固定周期或明确触发源。
- 必须能测最大耗时。
- 必须能统计超时。
- 必须没有动态内存。
- 必须没有文件系统。
- 必须没有阻塞等待。
- 必须没有无上限循环。
- 必须尽量少用临界区。

慢路径规则：

- 可以模块化。
- 可以使用队列、事件、状态机。
- 可以做协议解析。
- 可以写日志，但不能阻塞高频任务。
- 可以更新配置，但要生成快照给快路径使用。

## 8. 一条比较稳的落地顺序

建议按这个顺序推进：

1. 加实时性统计模块，只统计，不改变行为。
2. 跑一次 HERO 当前代码，记录高频任务最大耗时。
3. 把 `chassis_control_task` 改成 `vTaskDelayUntil`。
4. 清理高频任务里的重复调用和明显非必要工作。
5. 把日志和 watch 在高频任务里的成本降到最低。
6. 给 CAN 接收处理加时间预算或数量预算。
7. 把高频控制参数整理成快照。
8. 再决定是否合并成 `control_executor_task`。
9. 快路径稳定后，再做慢路径模块注册、配置服务、诊断中心。

## 9. 判断改造有没有成功

不要只看“能跑”，要看这些指标：

- 高频任务最大耗时有没有下降。
- 高频任务周期抖动有没有下降。
- CAN 命令从控制计算完成到发出的延迟有没有下降。
- 10 分钟压力测试有没有超时。
- 开启上位机、裁判系统、日志后，快路径耗时是否基本不变。
- 新加一个慢路径模块时，是否不会影响云台、底盘、射击控制周期。

如果这些指标没变好，说明架构改造只是变复杂了，没有真的提高实时性。

## 10. 当前建议结论

短期不要急着引入大框架。先把 ARBATOS 做成“快路径硬、慢路径软”的结构。

短期目标：

- 让现有多任务结构可测。
- 先修掉明显抖动源。
- 让 CAN 发送、云台、射击、底盘的时间关系更确定。

中期目标：

- 建立快照接口。
- 建立高频控制边界。
- 必要时引入统一控制执行器。

长期目标：

- 在慢路径里融合模块注册、配置服务、诊断中心、事件通知。
- 保持快路径不被高级框架污染。
- 让 ARBATOS 同时具备实时性和可扩展性。

# Task Family Architecture

现在这套改法，核心就一句话：

- 小差异，留在同一个任务家族里，改参数。
- 大差异，单开一个新任务家族。

当前已经加了一个显式配置入口：`g_config.profile`

它负责说明这块板默认起哪类高频任务：

- `locomotion_family`
- `gimbal_family`
- `arm_family`

## 现阶段建议

- `mecanum` / `xdrive`：算同一个移动家族，继续走 `chassis_control_task`
- 单云台：单独一个云台家族
- 双云台：单独一个云台家族
- 舵机轮腿：单独一个移动家族
- MIT 轮腿：再单独一个移动家族
- 机械臂如果控制拓扑明显不同，也单开新家族

## 开发者加新家族时怎么做

1. 在 `config.h` 的家族枚举里补一个新值。
2. 在对应 target 的 `g_config.profile` 里选这个家族。
3. 新建自己的任务文件。
4. 在板级 `freertos.c` 或 `board_freertos.c` 里加一条创建入口。
5. 给这个任务家族单独准备参数块。

## 现阶段不要做的事

- 不要再把底盘、云台、机械臂硬塞进一个统一高频总循环。
- 不要靠“某个电机 ID 不为 0”去猜系统该起什么任务。
- 不要指望只改参数就把一种大类机器人变成另一种。

## 这套东西解决的是什么

它不是让最终用户随便拼机器人。

它解决的是：开发者以后加一种新形态时，入口更固定，改动范围更小，默认任务也更清楚。

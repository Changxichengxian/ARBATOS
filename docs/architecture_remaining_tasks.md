# ARBATOS 剩余架构任务

这份只保留还没完成的内容。

## 实时控制剩余任务

1. 补真实运行数据。
   - 跑一次 HERO 主线，记录云台、底盘、CAN 发送、CAN 接收、`sdlog_write`、`watch_task_beat` 的最大耗时、平均耗时和超时次数。
   - 至少留一组正常比赛负载下的 10 分钟数据，后面改架构时有对比基线。

2. 收紧高频任务里的日志和观察成本。
   - 继续检查 1ms / 2ms 任务里的直接 `sdlog_write` 调用。
   - 高频任务里只允许极简计数和批量缓存，真正写文件放到低优先级任务里。
   - 给这条规则补成明确代码约束，避免后面又把文件写入塞回高频循环。

3. 做统一的快路径快照接口。
   - `sensor_snapshot_t`：姿态、角速度、IMU 状态。
   - `motor_feedback_snapshot_t`：电机反馈状态。
   - `input_snapshot_t`：遥控、图传、上位机输入整理后的结果。
   - `control_config_snapshot_t`：高频控制只读参数。
   - `actuator_cmd_snapshot_t`：本周期要发给执行器的命令。

4. 把高频配置读取改成快照读取。
   - 云台、底盘、射击这些高频任务不要每圈直接读一大坨 `g_config`。
   - 慢路径更新配置后，用短临界区切换快照。
   - 快路径只读已经准备好的结构。

5. 用数据决定要不要做统一控制执行器。
   - 如果现有多任务调度下，控制周期抖动或 CAN 发送延迟仍然明显，再做 `control_executor_task`。
   - 这个任务只负责固定顺序执行云台、射击、底盘和执行器命令发送。
   - 没有数据证明前，先不要把所有高频控制硬塞到一个总循环里。

6. 补慢路径服务。
   - 配置服务：集中处理参数校验、参数更新、快照切换。
   - 诊断服务：集中整理任务耗时、设备在线、错误码、电源状态。
   - 低频事件通知：给设备上线、模式切换、错误上报这类低频事件用。
   - 模块注册只放慢路径，不进入 1ms 控制链。

## 任务家族剩余任务

1. 补真正的双云台任务家族。
   - 现在有 `GIMBAL_FAMILY_DUAL`、配置块和观察块，但没有对应的双云台控制任务。
   - 需要明确双云台的数据结构、控制任务入口、默认参数和板级创建条件。

2. 补舵机轮腿任务家族。
   - 现在有 `LOCOMOTION_FAMILY_WHEELLEG_SERVO`、配置块和观察块，但没有实际控制任务。
   - 需要定义轮腿状态、输入映射、控制周期、执行器输出和安全停机逻辑。

3. 补 MIT 轮腿任务家族。
   - 现在有 `LOCOMOTION_FAMILY_WHEELLEG_MIT`、配置块和观察块，但没有实际控制任务。
   - 需要明确 MIT 电机反馈、命令发送、限位、失联保护和调试观察字段。

4. 把新家族接进板级任务创建。
   - `robot_profile_need_*` 里已经有判断入口，但 `freertos.c` / `board_freertos.c` 还没有创建双云台、舵机轮腿、MIT 轮腿任务。
   - 新任务实现后，要在各工程入口里按 profile 创建。

5. 给新家族补完整参数块。
   - 现在这些块更像占位。
   - 等实际控制任务确定后，再补 PID、周期、限幅、输入映射、在线检测、调试观察字段。

6. 补新家族的最小验证方式。
   - 至少要能确认：profile 选中后只创建对应任务。
   - 不该启动的家族任务必须保持关闭。
   - 新家族离线或安全档时，执行器输出必须清零或进入安全阻尼。
